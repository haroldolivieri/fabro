title: "Portkey AI Gateway"
description: "Route Fabro's LLM traffic through Portkey for observability, cost tracking, and provider routing"
---

Fabro can route all LLM requests through [Portkey](https://portkey.ai), an AI gateway that provides observability, cost tracking, fallback routing, and access to providers like AWS Bedrock and Azure OpenAI. When enabled, Portkey sits between Fabro and your LLM provider — Fabro sends requests to Portkey's gateway, and Portkey forwards them to the configured upstream provider.

No code changes are needed. Set three environment variables and Fabro routes through Portkey automatically.

## Prerequisites

- A [Portkey](https://portkey.ai) account with an API key
- At least one provider configured in your [Portkey Model Catalog](https://app.portkey.ai/model-catalog)

## How it works

Portkey uses **provider slugs** (`@openai-prod`, `@bedrock-sandbox`, etc.) to identify your configured integrations in the [Model Catalog](https://app.portkey.ai/model-catalog). When you set `PORTKEY_PROVIDER_SLUG`, Fabro sends that slug as the `x-portkey-provider` header, and Portkey routes the request to the right integration using its stored credentials.

### Native provider format (optional)

By default, Fabro sends requests in **OpenAI-compatible format**, which Portkey accepts for all providers. This is sufficient for most use cases.

Set `PORTKEY_PROVIDER` if you want to use the **provider's native API format** and unlock provider-specific features:

| Provider | Native features unlocked |
|----------|-------------------------|
| `anthropic` | Prompt caching, extended thinking, interleaved reasoning |
| `openai` | Responses API, built-in tools, reasoning tokens |
| `gemini` | Native Gemini features |

Leave `PORTKEY_PROVIDER` unset when you don't need these features, or when your slug routes to a provider that differs from any of the above (e.g. a custom integration).

## Setup paths

### Path A: Any provider via Model Catalog (recommended)

The simplest setup — Portkey handles all authentication and routing via the slug.

```bash title=".env"
PORTKEY_URL=https://api.portkey.ai/v1
PORTKEY_API_KEY=pk-xxx
PORTKEY_PROVIDER_SLUG=@openai-prod
```

Use standard model IDs in your workflow stylesheet:

```css
* { model: gpt-4o; }
```

Find your slugs at [app.portkey.ai/model-catalog](https://app.portkey.ai/model-catalog).

### Path B: AWS Bedrock via Model Catalog

Route requests to AWS Bedrock. Set `PORTKEY_PROVIDER=anthropic` to use the native Anthropic Messages API format (required for Bedrock Claude models, and to unlock caching/thinking).

```bash title=".env"
PORTKEY_URL=https://api.portkey.ai/v1
PORTKEY_API_KEY=pk-xxx
PORTKEY_PROVIDER_SLUG=@bedrock-sandbox
PORTKEY_PROVIDER=anthropic
```

**Model IDs must use Bedrock inference profile format** — region-prefixed identifiers:

```css
* { model: eu.anthropic.claude-sonnet-4-6; }
.code { model: eu.anthropic.claude-opus-4-6-v1; }
```

Common Bedrock inference profile IDs for Claude models:

| Model | EU | US | Global |
|-------|----|----|--------|
| Claude Sonnet 4.6 | `eu.anthropic.claude-sonnet-4-6` | `us.anthropic.claude-sonnet-4-6` | `global.anthropic.claude-sonnet-4-6` |
| Claude Opus 4.6 | `eu.anthropic.claude-opus-4-6-v1` | `us.anthropic.claude-opus-4-6-v1` | `global.anthropic.claude-opus-4-6-v1` |
| Claude Haiku 4.5 | `eu.anthropic.claude-haiku-4-5-20251001-v1:0` | `us.anthropic.claude-haiku-4-5-20251001-v1:0` | `global.anthropic.claude-haiku-4-5-20251001-v1:0` |
| Claude Opus 4.5 | `eu.anthropic.claude-opus-4-5-20251101-v1:0` | `us.anthropic.claude-opus-4-5-20251101-v1:0` | `global.anthropic.claude-opus-4-5-20251101-v1:0` |

Full list: [AWS Bedrock cross-region inference profiles](https://docs.aws.amazon.com/bedrock/latest/userguide/inference-profiles-support.html)

### Path C: AWS Bedrock with direct AWS credentials

Route to Bedrock without a Model Catalog provider — pass AWS credentials via Portkey headers instead.

```bash title=".env"
PORTKEY_URL=https://api.portkey.ai/v1
PORTKEY_API_KEY=pk-xxx
PORTKEY_PROVIDER_SLUG=@bedrock-sandbox
PORTKEY_PROVIDER=anthropic
PORTKEY_AWS_ACCESS_KEY_ID=AKIA...
PORTKEY_AWS_SECRET_ACCESS_KEY=xxx
PORTKEY_AWS_REGION=eu-west-1
# PORTKEY_AWS_SESSION_TOKEN=    # Optional: for STS/assumed-role access
```

Uses the same Bedrock inference profile model IDs as Path B.

### Path D: Config-based routing (fallbacks, load balancing)

Use a [Portkey config](https://app.portkey.ai/configs) for advanced routing strategies: fallbacks, load balancing, or conditional routing across providers.

```bash title=".env"
PORTKEY_URL=https://api.portkey.ai/v1
PORTKEY_API_KEY=pk-xxx
PORTKEY_PROVIDER_SLUG=@openai-prod
PORTKEY_CONFIG=cfg-xxx
```

The slug is still required — it serves as the default target when the config does not override routing for a given request.

## Environment variable reference

| Variable | Required | Description |
|----------|----------|-------------|
| `PORTKEY_URL` | Yes | Gateway base URL. For hosted Portkey: `https://api.portkey.ai/v1`. For self-hosted gateways, use your gateway URL with `/v1` appended. |
| `PORTKEY_API_KEY` | Yes | Your Portkey API key. [Get one here](https://app.portkey.ai/api-keys). |
| `PORTKEY_PROVIDER_SLUG` | Yes | Model Catalog slug (e.g. `@openai-prod`). Sent as the `x-portkey-provider` header. [Find yours here](https://app.portkey.ai/model-catalog). |
| `PORTKEY_PROVIDER` | No | Set to use the provider's native API format and unlock provider-specific features (e.g. `anthropic` for prompt caching and extended thinking). When unset, OpenAI-compatible format is used. Valid values: `anthropic`, `openai`, `gemini`, `kimi`, `zai`, `minimax`, `inception`. |
| `PORTKEY_CONFIG` | No | Config ID for advanced routing (fallbacks, load balancing). [Manage configs](https://app.portkey.ai/configs). |
| `PORTKEY_AWS_ACCESS_KEY_ID` | No | AWS access key for direct Bedrock access (Path C). |
| `PORTKEY_AWS_SECRET_ACCESS_KEY` | No | AWS secret key. |
| `PORTKEY_AWS_REGION` | No | AWS region (defaults to `us-east-1`). |
| `PORTKEY_AWS_SESSION_TOKEN` | No | AWS STS session token for federated/assumed-role access. |
| `PORTKEY_METADATA` | No | JSON metadata for Portkey observability (e.g. `{"team":"eng"}`). Set via environment variable only. |

## Known limitations

- **All-or-nothing routing.** When Portkey is configured, all requests go through Portkey. For selective routing, use `PORTKEY_CONFIG` with conditional routing rules.
- **Single slug per configuration.** To route different models to different providers, use a `PORTKEY_CONFIG` that handles routing internally.
- **No model ID translation.** Fabro passes model IDs verbatim. When targeting Bedrock, use Bedrock inference profile IDs, not standard Anthropic model names.

## Troubleshooting

**Requests fail with 401 from Portkey** — Verify `PORTKEY_API_KEY` is correct and `PORTKEY_PROVIDER_SLUG` matches a slug in your [Model Catalog](https://app.portkey.ai/model-catalog).

**`PORTKEY_PROVIDER` is not a recognized provider name** — The value must be one of: `anthropic`, `openai`, `gemini`, `kimi`, `zai`, `minimax`, `inception`. Check for typos.

**Requests fail with "model not found" from Bedrock** — You're likely using a standard Anthropic model ID instead of a Bedrock inference profile ID. Use the region-prefixed format: `eu.anthropic.claude-sonnet-4-6`.

**Requests fail with 403 from Bedrock** — Your AWS credentials lack the `bedrock:InvokeModel` and `bedrock:GetInferenceProfile` IAM permissions.

**Portkey env vars are set but requests go directly to the provider** — `PORTKEY_URL`, `PORTKEY_API_KEY`, and `PORTKEY_PROVIDER_SLUG` must all be set. If any is missing, Portkey integration is silently disabled.
