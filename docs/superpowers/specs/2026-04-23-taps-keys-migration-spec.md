# taps-keys Migration: Three-Repo Architecture

## 1. Problem Statement

Skyscanner's flight pricing keys (taps-keys) are encoded by a Java library that defines 142 key schemas (71 one-way, 71 return). Each schema maps a flight route + date + directionality combination into a compact base-32 encoded string used as a lookup key in the File Cache Proxy (FCP) — Skyscanner's distributed cache for aggregated pricing data.

The `quote-aggregator` Java Spark pipeline is being migrated to Python (SkySpark). The `taps-keys` library is the highest-risk dependency: both the Java and Python pipelines must produce **byte-for-byte identical** keys for the same inputs. A single wrong character means a cache miss or, worse, data written to the wrong FCP file store.

A single source of truth for schema definitions is needed to prevent drift between Java and Python implementations. Without it, a schema added to one language but forgotten in the other, or a component order changed inconsistently, would produce silent data corruption.

## 2. Architecture

Three repos, each with a distinct responsibility:

| Repo | Purpose | Consumers |
|---|---|---|
| **taps-keys-schemas** | Single source of truth for schema definitions (`schemas.json`). Structural validation. Claude Code harness for managing schema additions. | Java taps-keys, Python taps-keys (runtime) |
| **taps-keys-fixtures** | Golden encoding/signature fixtures generated from the Java library. Contract test runner that validates any implementation against fixtures. | CI pipelines (test-time only) |
| **taps-keys-python** | Python implementation of the encoding library. Reads schemas.json from taps-keys-schemas at runtime. | Python pipelines (quote-aggregator, SkySpark) |

## 3. Key Design Decisions

| Decision | Rationale |
|---|---|
| **Fixtures generated from real Java library** | The Java library is the existing production system. Fixtures generated by running it are the immutable ground truth — no agent extraction can introduce errors in the golden data. |
| **Schemas exported from compiled Java library (SchemaExporter), validated against fixtures** | `SchemaExporter` uses reflection on `Keys.OneWay`/`Keys.Return` to export schema metadata (name, prefix, to_string, encoded_length, open_jaw_filter) from the same compiled library version as the golden fixtures. This eliminates source/binary version skew and removes agent hallucination risk. The agent parses `to_string` to reconstruct the component list. |
| **Python coding agent has NO access to fixtures** | True isolation via package separation. The Python agent implements the algorithm from a spec and worked examples, not by pattern-matching against test data. This is the sf-try methodology: seed with understanding, validate against hidden oracle. |
| **Contract runner reveals mismatch type but never full expected values** | Error output shows which schema failed, the nature of the mismatch (length difference, first differing position), and at most 4 characters of the expected value. This gives enough to debug without leaking the answer. |
| **Both libraries consume schemas.json at runtime** | A single definition file eliminates drift by construction. Neither language owns the schema definitions — both are downstream consumers. |
| **Fixture generation only needed for migration, not for adding new schemas** | Once the Python encoding algorithm is proven correct against all 142 schemas x 5 input sets, adding a new schema only requires updating schemas.json. The proven algorithm handles it automatically. |
| **Three repos instead of one monorepo** | Clean separation of concerns. The schemas repo has no test infrastructure. The fixtures repo has no encoding logic. The Python repo has no golden data. Each team owns what they need. |
| **KeySchema must be picklable** | PySpark ships UDF closures to executor nodes via pickle. A non-picklable KeySchema would fail at runtime in Spark. |
| **Replicate Java bugs exactly** | YEARMONTH value 1024 (April 2055) silently overflows the 2-char encoding width. The Python port must reproduce this — "fixing" it would break byte-for-byte compatibility. |

## 4. Migration Workflow (fabro)

The migration is automated as a fabro workflow with 3 phases, 19 nodes, 3 human review gates, and fixture isolation enforced by package separation.

### Phase 1: Build taps-keys-fixtures

**Input**: Pre-generated golden fixture JSON files (from running the real Java library), `Keys.java` source.

**Agent task**: Copy fixtures into package structure. Build the contract test runner (`test_runner.py`, `error_formatter.py`) that can validate any Python taps-keys implementation against the golden data. Build self-tests for the runner infrastructure. Wire up repo harness (pyproject.toml, Makefile, README, CLAUDE.md).

**Validation gate**: `pip install -e .` succeeds. `pytest tests/ -v` passes (self-tests). Runner CLI produces correct error on nonexistent module. Error formatter never leaks full expected values.

**Flow**: Agent builds -> Validate -> Pass? -> Human review -> Publish to Artifactory

### Phase 2: Build taps-keys-schemas

**Input**: `/tmp/schema_seed.json` (output of `SchemaExporter` Java tool, run against the fat JAR in setup). Golden fixtures (for cross-ref validation).

**Agent task**: Read schema seed, parse `to_string` parts to reconstruct component lists, build `schemas.json`. Build structural validation (`validate.py`, `schemas.schema.json`). Build Claude Code harness (`.claude/agents/add-schema.md` skill). Wire up repo harness.

**Validation gate**: 13 structural checks pass (JSON Schema conformance, schema count = 71+71, valid component types, no duplicates, toString derivation, encoded_length consistency, OpenJawFilter derivation, fixture coverage 142 x 5 = 710 entries, bidirectional coverage, signature consistency, toString consistency).

**Flow**: Agent extracts -> Validate (13 checks) -> Pass? -> Human review -> Publish to Artifactory

### Phase 3: Build taps-keys-python

**Input**: Encoding seed document (algorithm spec with 10 worked examples). `schemas.json` via `taps-keys-schemas` package. NO Java source. NO fixture data.

**Agent task**: Implement the full Python library (`encoding.py`, `schema.py`, `keys.py`, `website_id.py`). Write unit tests. Wire up repo harness. Build `scripts/validate.py` that invokes the contract runner from taps-keys-fixtures.

**Validation gate**: Contract runner passes all layers (L1: encoding correctness across 710 cases, L2: signature/disjoint correctness across 852 checks, L3: structural properties). Agent's own unit tests pass (Layer 4).

**Flow**: Agent implements -> Contract runner validates -> Pass? -> Human review -> Done

### Retry and Escalation

Each phase has a retry loop with controlled error feedback:
- Phase 1: max 3 retries (mechanical task)
- Phase 2: max 3 retries (extraction is deterministic)
- Phase 3: max 5 retries for encoding, max 3 retries for harness (algorithm implementation may need iteration)

On exhausted retries, the workflow escalates to a human stage. Error output from each retry is appended to the agent's context so it can see what it tried and what failed.

The 5 input sets per schema provide input diversity:

| Set | Origin | Dest | Carrier | Outbound | Inbound | isDirect | Purpose |

## 6. Encoding Algorithm

### Base-32

Alphabet: `0123456789abcdefghijklmnopqrstuv` (equivalent to Java's `Integer.toString(n, 32)`). Always lowercase.

Encoding: repeatedly divmod by 32, map remainders to alphabet, left-pad with `'0'` to fixed width.

### Component Types

| Type | bits | encoded_length (chars) | Value range | Encoding formula |
|---|---|---|---|---|
| AIRPORT | 16 | 4 | [0, 65536] | direct base-32 |
| CITY | 16 | 4 | [0, 65536] | direct base-32 |
| COUNTRY | 16 | 4 | [0, 65536] | direct base-32 |
| LOCATION | 16 | 4 | [0, 65536] | direct base-32 |
| YEARMONTH | 10 | 2 | [0, 1024] | `(year - 1970) * 12 + (month - 1)` |
| DAY | 5 | 1 | [0, 32] | day-of-month directly |
| DIRECT | 1 | 1 | {0, 1} | `true -> 1, false -> 0` |
| MARKETING_CARRIER | 16 | 4 | [-32768, 32767] | `carrier_id + 32768`, then base-32 |

### Key Encoding

`Key.encode()` = concatenate each component's base-32 string in schema order. Wildcards are trailing-only: stop appending at first wildcard.

### YEARMONTH Overflow

Value 1024 (April 2055) produces `"100"` (3 chars) for a nominally 2-char field. This is a latent edge case in Java — the range check passes (`<= 2^10`) but the encoded result exceeds the nominal width. Both implementations must replicate this exact behaviour.

## 7. Adding a New Schema (Post-Migration)

Once the encoding algorithm is proven correct against all 142 schemas, adding a new schema requires NO fixture regeneration:

1. **Update schemas.json** in `taps-keys-schemas` — add the new schema's component list
2. **Run structural validation** (`validate.py`) in taps-keys-schemas — verifies JSON structure, component types, toString derivation, encoded_length consistency
3. **Both Java and Python pick it up automatically** — both libraries load schemas.json at runtime. The encoding algorithm is proven correct by the 710 existing fixture cases; a new schema with the same component types will encode correctly by construction.
4. **No fixture regeneration needed** — the fixtures proved the algorithm, not individual schemas. New schemas use the same proven algorithm.
5. **Claude Code `add-schema` skill** in taps-keys-schemas guides the process — prompts for prefix and components, appends to schemas.json, runs validation.

For downstream consumers:
- `pip install --upgrade taps-keys-schemas` in the Python repo
- The Java repo's build pulls the latest schemas.json
- Both CI pipelines verify the new schema loads and encodes without error

## 9. Repo Ownership

| Repo | Owner | Rationale |
|---|---|---|
| **taps-keys-schemas** | Pricing Platform team (shared) | Schemas are the shared contract between Java and Python. Changes affect both pipelines. The owning team must coordinate across consumers. |
| **taps-keys-fixtures** | Pricing Platform team (shared) | Fixtures validate both implementations. The owning team maintains the contract runner and ensures fixture integrity. |
| **taps-keys-python** | Python Pipeline team | The Python implementation is owned by the team building and operating the Python pipelines (SkySpark, quote-aggregator migration). |

### Collaboration Model

- **Schema changes** require a PR to taps-keys-schemas, reviewed by the Pricing Platform team. Both Java and Python CI pipelines run against the updated schemas before merge.
- **Fixture changes** are rare (only needed if the Java library's behaviour changes). Requires re-running the Java fixture generator and a PR reviewed by Pricing Platform.
- **Python implementation changes** are reviewed by the Python Pipeline team. The contract runner (from taps-keys-fixtures) validates correctness automatically in CI.

## 11. Production-Critical Features

These features are exercised by `quote-aggregator` in production. Incorrect implementation causes data corruption:

| Feature | How quote-aggregator uses it | Risk if wrong |
|---|---|---|
| `Key.encode()` | Every quote row gets a key via Spark UDF | Wrong keys = FCP lookups return wrong prices |
| `KeySchema.toString()` | Used as FCP file store name | Wrong toString = writes to wrong file store, data loss |
| `KeySchema.signature()` + disjoint | Determines which fields go in key vs protobuf value | Wrong disjoint = missing fields or redundant data |
| `OpenJawFilter` | Drives SQL WHERE clauses for return flights | Wrong filter = missing or duplicate return flight data |
| `encode_to_fixed32(string)` | Encodes website ID for FCP filter keys | Wrong encoding = FCP filter lookups fail |

## 12. Future State: Java Refactor

The three-repo architecture is designed to support a future Java refactor where `Keys.java` is generated from `schemas.json` rather than hardcoded:

### Current State (this migration)

```
taps-keys-schemas/
  schemas.json  <-- extracted from Java Keys.java (one-time)
       |
       +------> taps-keys-python reads at runtime

taps-keys (Java)
  Keys.java     <-- still hardcoded, untouched
```

### Future State

```
taps-keys-schemas/
  schemas.json  <-- the ONLY place schemas are defined
       |
       +------> taps-keys-python reads at runtime
       |
       +------> taps-keys (Java) generates Keys.java via Gradle codegen
```

The Java refactor is not blocked on the Python port. The sequence is:

1. **Now**: Build three repos + Python lib (this spec)
2. **Later**: Add Gradle codegen task to Java lib that reads schemas.json
3. **Later**: Delete hardcoded Keys.java constants, replace with generated code
4. **Later**: Add CI to Java repo that pulls schemas.json from taps-keys-schemas

Steps 2-4 can happen independently at whatever pace the Java team is comfortable with. The schemas repo is ready for it from day one.
